# スキル: ガベージコレクション

> **実行タイミング**:
> - 週次（毎週月曜日など定期的に）
> - 大きな機能マージ後
> - ドキュメントと実装の乖離が疑われるとき
>
> **目的**: エージェントが生成したコードやドキュメントのドリフトを定期的に解消し、
> リポジトリの健全性を継続的に維持する。
>
> **所要時間の目安**: リポジトリ規模によって異なるが、1回の実行で30分以内を目標とする。

---

## このスキルがすること

以下を順番に実行する。

```
① ドキュメント鮮度スキャン（全件）
② アーキテクチャドリフト検出（全件）
③ ドキュメントライフサイクル整理
④ CONTEXT.md 更新
⑤ GC レポート生成
```

---

## 手順

### ① ドキュメント鮮度スキャン（check-doc-freshness 全件モード）

`check-doc-freshness` スキルをリポジトリ全体に対して実行する。

- 対象: `docs/**/*.md` 内の `tracks:` フィールドを持つすべてのドキュメント
- `tracks:` のパターンに対応するコードファイルを読み込み、乖離を確認する
- 乖離が見つかった場合は修正する

**合わせて確認すること:**
- `tracks:` フィールドが設定されていないドキュメントに対して、設定が必要かどうかを判断する
  - コードと1:1で対応するドキュメントには `tracks:` を追加する
  - 設定が不要な場合はスキップする

### ② アーキテクチャドリフト検出（check-invariants 全件モード）

`check-invariants` スキルを `src/**` 全体に対して実行する。

- `invariants.md` のすべての INV に対して、対応する全ファイルを確認する
- 違反が見つかった場合は修正する
- 「警告」レベルの違反が多い場合は、`invariants.md` の条件見直しも検討する

### ③ ドキュメントライフサイクル整理

`document_architecture.md` §8.2 の `status:` 遷移ルールに従い、各ドキュメントの状態を確認する。

| 確認内容 | 判断基準 | アクション |
|---------|---------|----------|
| `draft` が7日以上経過 | `created:` 日付から判断 | `active` に更新するか、理由を記録して継続 |
| `active` だが対応コードが削除されている | `tracks:` ファイルが存在しない | `deprecated` に変更 |
| `deprecated` が30日以上経過 | `status: deprecated` 日付から判断 | 削除（または確認を促す） |
| `exec-plans/active/` の受け入れ条件がすべて完了 | チェックリストを確認 | `complete-exec-plan` の実行を提案 |

### ④ CONTEXT.md 更新

`update-context` スキルを実行する。

- 現在フェーズが正確かどうかを確認・更新する
- 優先タスクが最新の `exec-plans/active/` を反映しているかを確認・更新する

### ⑤ GC レポート生成

```markdown
## GC レポート YYYY-MM-DD

### ① ドキュメント鮮度
- チェックしたドキュメント: {件数}件
- 更新が必要だった（対応済み）: {件数}件
  - {ドキュメントパス}: {更新内容}
- 未対応（手動確認が必要）: {件数}件

### ② アーキテクチャドリフト
- チェックした INV: {件数}件
- チェックしたファイル: {件数}件
- 違反（修正済み）: {件数}件
- 警告: {件数}件

### ③ ライフサイクル整理
- draft → active: {件数}件
- active → deprecated: {件数}件
- 削除: {件数}件

### ④ CONTEXT.md
- 更新内容: {変更点}

### 総合評価
{問題なし / 軽微な修正を実施 / 要注意事項あり}
```

レポートは `exec-plans/completed/YYYY-MM-gc.md` に記録する（GC 実行の履歴として残す）。
