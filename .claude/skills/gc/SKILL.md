# Skill: Garbage Collection

> **When to run**:
> - Weekly (e.g., every Monday on a regular schedule)
> - After a large feature merge
> - When documentation and implementation are suspected to have drifted
>
> **Purpose**: Periodically resolve drift in code and documentation generated by agents,
> and continuously maintain the health of the repository.
>
> **Estimated time**: Varies with repository size; target completion within 30 minutes per run.

---

## What this skill does

Runs the following steps in order.

```
① Document freshness scan (all documents)
② Architecture drift detection (all files)
③ Document invariant check (all documents)
④ Document lifecycle cleanup
⑤ CONTEXT.md update
⑥ GC report generation
```

---

## Steps

### ① Document freshness scan (check-doc-freshness full-scan mode)

Run the `check-doc-freshness` skill against the entire repository.

- Target: all documents in `docs/**/*.md` that have a `tracks:` field
- Load code files matching the `tracks:` patterns and verify for discrepancies
- Fix any discrepancies found

**Also check:**
- For documents that do not have a `tracks:` field, determine whether one should be added
  - Add `tracks:` to documents that have a 1:1 correspondence with code
  - Skip if not necessary

### ② Architecture drift detection (check-invariants full-scan mode)

Run the `check-invariants` skill against all of `src/**`.

- Check all files corresponding to every INV in `invariants.md`
- Fix any violations found
- If there are many "warning"-level violations, consider revising the conditions in `invariants.md`

### ③ Document invariant check (check-doc-invariants full-scan mode)

Run the `check-doc-invariants` skill against all documents in the repository.

- Target: all `docs/**/*.md` and `exec-plans/**/*.md`
- Check DOC-INV-001 (reference direction), DOC-INV-002 (frontmatter completeness),
  DOC-INV-003 (lifecycle consistency), DOC-INV-004 (AC traceability),
  DOC-INV-005 (diagram rules)
- Fix any violations found
- For DOC-INV-005 warnings: use context to decide whether Mermaid conversion is worthwhile;
  record the decision if leaving ASCII art intentionally

### ④ Document lifecycle cleanup

Review the status of each document according to the following `status:` transition rules.

| Transition | Trigger |
|------------|---------|
| (new) → `draft` | On file creation |
| `draft` → `active` | On PR merge, or when reviewed within 7 days of creation |
| `active` → `deprecated` | When the corresponding code/feature is deleted, or when a successor document becomes `active` |
| `deprecated` → (deleted) | Can be deleted 30 days after becoming `deprecated` |


| Check item | Criteria | Action |
|-----------|----------|--------|
| `draft` older than 7 days | Determined from `created:` date | Update to `active`, or continue with a recorded reason |
| `active` but corresponding code deleted | `tracks:` files do not exist | Change to `deprecated` |
| `deprecated` older than 30 days | Determined from `status: deprecated` date | Delete (or prompt for confirmation) |
| All acceptance criteria in `exec-plans/active/` are complete | Review checklist | Suggest running `complete-exec-plan` |

### ⑤ CONTEXT.md update

Run the `update-context` skill.

- Verify and update the current phase for accuracy
- Verify and update the priority tasks to reflect the latest `exec-plans/active/`

### ⑥ GC report generation

```markdown
## GC Report YYYY-MM-DD

### ① Document freshness
- Documents checked: {count}
- Updates needed (addressed): {count}
  - {document path}: {update description}
- Not addressed (manual review needed): {count}

### ② Architecture drift
- INVs checked: {count}
- Files checked: {count}
- Violations (fixed): {count}
- Warnings: {count}

### ③ Document invariants
- Documents checked: {count}
- DOC-INV-001 violations (reference direction): {count}
- DOC-INV-002 violations (frontmatter): {count}
- DOC-INV-003 violations (lifecycle): {count}
- DOC-INV-004 violations (AC traceability): {count}
- DOC-INV-005 warnings (diagram rules): {count}

### ④ Lifecycle cleanup
- draft → active: {count}
- active → deprecated: {count}
- Deleted: {count}

### ⑤ CONTEXT.md
- Changes: {description of changes}

### Overall assessment
{No issues / Minor fixes applied / Items requiring attention}
```

Record the report in `exec-plans/completed/YYYY-MM-gc.md` (kept as a history of GC runs).
