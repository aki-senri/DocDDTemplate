# DocDD 開発ルール

このプロジェクトはドキュメント駆動開発（DocDD）を採用しています。
実装はすべて仕様（exec-plan の AC）に基づいて行います。

## 仕様ファースト原則（必須）

**実装の指示を受けたとき、コードを書く前に必ず以下を確認する。**

### 確認チェックと応答

| 状態 | Claudeの応答 |
|------|-------------|
| `exec-plans/active/` に実行計画なし | `/create-exec-plan` を提案し、実装しない |
| exec-plan に `AC-001〜` の定義なし | AC の追加を求め、実装しない |
| AC はあるが指示に AC 番号なし | 「どの AC を実装しますか？」と確認し、実装しない |
| AC 番号が指定されている | 実装を開始する |

### 応答例

**exec-plan なしの場合:**
```
仕様（exec-plan）が定義されていません。
先に /create-exec-plan を実行して受け入れ基準（AC）を定義しましょう。
```

**AC が指定されていない場合:**
```
どの受け入れ基準（AC）を実装しますか？
現在の exec-plan: exec-plans/active/YYYY-MM-xxx.md
  - AC-001: ...
  - AC-002: ...
```

## 例外処理（仕様なしで進める場合）

ユーザーが「仕様なしで進めることを確認します」などの明示的な確認を与えた場合のみ：

1. `exec-plans/.spec-override` を作成する（理由とタイムスタンプを記録）
2. 実装を進める
3. 実装完了後に `exec-plans/.spec-override` を削除する

## テスト変更の禁止

テストを修正する場合は対応する AC-ID を必ず確認すること。
実装の挙動に合わせたテスト修正は禁止。変更理由を exec-plan の Decision Log に記録する。

## スキル一覧

| スキル | 目的 | タイミング |
|--------|------|-----------|
| `/create-exec-plan` | AC（受け入れ基準）を定義する | 実装前（必須） |
| `/start-feature` | 実装開始前の準備チェック | create-exec-plan 後 |
| `/run-tests` | テスト実行と仕様照合 | 実装中・完了時 |
| `/check-doc-freshness` | コードとドキュメントの乖離確認 | コード変更後 |
| `/check-invariants` | INV 違反確認 | コード変更後 |
| `/pre-pr` | PR 前の総合チェック | PR 作成前（必須） |
| `/complete-exec-plan` | exec-plan を完了状態に移動 | PR マージ後 |
| `/gc` | 全体的なドリフト解消 | 週次 |
