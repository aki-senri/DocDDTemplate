# DocDD 開発ルール

このプロジェクトはドキュメント駆動開発（DocDD）を採用しています。
実装はすべて仕様（exec-plan の AC）に基づいて行います。

## 仕様ファースト原則（必須）

**実装の指示を受けたとき、コードを書く前に必ず以下を確認する。**

### 確認チェックと応答

| 状態 | Claudeの応答 |
|------|-------------|
| `exec-plans/active/` に実行計画なし | `/create-exec-plan` を提案し、実装しない |
| exec-plan に `AC-001〜` の定義なし | AC の追加を求め、実装しない |
| AC はあるが指示に AC 番号なし | 「どの AC を実装しますか？」と確認し、実装しない |
| AC 番号が指定されている | 実装を開始する |

### 応答例

**exec-plan なしの場合:**
```
仕様（exec-plan）が定義されていません。
先に /create-exec-plan を実行して受け入れ基準（AC）を定義しましょう。
```

**AC が指定されていない場合:**
```
どの受け入れ基準（AC）を実装しますか？
現在の exec-plan: exec-plans/active/YYYY-MM-xxx.md
  - AC-001: ...
  - AC-002: ...
```

## 例外処理（仕様なしで進める場合）

ユーザーが「仕様なしで進めることを確認します」などの明示的な確認を与えた場合のみ：

1. `exec-plans/.spec-override` を作成する（理由とタイムスタンプを記録）
2. 実装を進める
3. 実装完了後に `exec-plans/.spec-override` を削除する

## ドキュメント内の図・ダイアグラム規則

ドキュメントに図を記載する際は以下のルールに従う。

| 状況 | ルール |
|------|--------|
| フロー・シーケンス・クラス図など | **Mermaid を優先して使用する** |
| Mermaid で表現できない図（UI スケッチ・レイアウト等） | AA（アスキーアート）を使用し、**直後に必ず図の説明文を添える** |

**AA に必ず添える説明文の例:**

```
┌──────────┬──────────┐
│ ファイル名 │ タグ     │
└──────────┴──────────┘
```

上図はファイル一覧画面のレイアウト。左列にファイル名、右列に付与済みタグを表示する。
行を選択するとタグ編集パネルが右側にスライドインする。

## テスト変更の禁止

テストを修正する場合は対応する AC-ID を必ず確認すること。
実装の挙動に合わせたテスト修正は禁止。変更理由を exec-plan の Decision Log に記録する。

## 仕様バージョンの管理（ブランチ運用）

仕様の更新は「現バージョンの修正」と「次バージョンの変更」を分離して管理する。
これにより、実装着手中に対象とする仕様が動くことを防ぎ、追従の判断を制御された地点（昇格）に集約する。

| 対象 | 置き場所 | 説明 |
|------|----------|------|
| 現在の開発ターゲット仕様 ＋ コード | `main` | 実装が *今* 追従すべき凍結された目標。`exec-plans/active/` は常にこの仕様に対応する |
| 次バージョンの仕様変更（docs のみ） | `spec/<label>` ブランチ | 例 `spec/sprint-12`。スプリント単位で作成。コードや exec-plan は含めない |

### 変更のルーティング（どちらに書くか）

| 変更種別 | 行き先 | 扱い |
|----------|--------|------|
| **現バージョンの不備修正**（仕様の誤り・矛盾・曖昧さの是正。スコープを広げない） | `main` に直接 | in-flight 作業に伝播させる（意図どおり）。既に実装済みの AC が影響を受ける場合は「stale」として reconcile 対象にする |
| **次バージョンの変更**（新機能・スコープ拡張・方針転換） | `spec/<label>` | 昇格まで現ターゲットを揺らさない |

判断基準は「これは『今作っている物が間違っている』のか、『次に作りたい物』なのか」。
ルーティングの判断とその理由は、対応する exec-plan の Decision Log に必ず記録する。

### 現バージョン修正による stale の扱い

`main` への不備修正が **既に実装済み** の AC の仕様を変える場合、その実装は新しい仕様に対して stale になる。
「なぜ stale にしたか」（履歴）と「reconcile すべき未処理作業」を別チャネルで管理する。

| 対象 AC の状態 | 対応 |
|----------------|------|
| **実装中**（`exec-plans/active/` にある） | そのプランの Decision Log に変更理由を記録する。作業は既に開いているため追加のプランは不要 |
| **完了済み**（`exec-plans/completed/` にある） | ① reconcile プランに同じ AC-ID を再オープンする（`active/` 内の最新の `*-reconcile.md` に追記、なければ当月の `exec-plans/active/YYYY-MM-reconcile.md` を作成）。② その **reconcile プラン側** の Decision Log に「AC-XXX は `<commit>` の不備修正で stale」と記録する（完了済みプランは archive 済みで参照されないため、そこには書かない） |

- 未処理の reconcile は必ず `exec-plans/active/` に置く（spec-gate・pre-pr・gc が走査する唯一の場所）。
- 専用の「Spec Drift Log」節は作らない（走査されず archive されて陳腐化するため）。
- reconcile プランは無限に追記せず、サイクルごとに閉じて新規を起こす（全 AC が `- [x]` にならず complete できなくなるのを防ぐ）。命名は in-version が `YYYY-MM-reconcile.md`、昇格起点（`/promote-spec`）が `YYYY-MM-reconcile-<label>.md`。両者は `active/` に併存し、扱い（同 AC-ID 再オープン＋Decision Log 記録）は同一。
- 当面 reconcile しないと判断した場合は、その active プランの Decision Log に `reconcile: deferred`（または `reconcile: n/a`）と記録する。これにより `/gc` backstop の再フラグを抑止する。
- 起票漏れは `/gc` の spec-drift backstop が週次で検出する（結果は確認候補であり確定ではない）。

### 昇格（promotion）

次バージョン仕様を現ターゲットに取り込む操作は **`/promote-spec`** で行う。
昇格は開発ターゲットを変える判断のため、**マージするか・いつするかは人が決める**（スキルは差分・影響解析と後処理を支援するのみ）。
昇格時に旧バージョンは `spec-target-<label>` タグで保存され、いつでも復元・差分比較できる。

## スキル一覧

| スキル | 目的 | タイミング |
|--------|------|-----------|
| `/create-requirements` | User Story・制約・AC を定義する | 実装前・create-exec-plan の前 |
| `/doc-review` | 独立エージェントによるドキュメントレビュー | create-requirements 後・create-exec-plan の前（任意） |
| `/create-exec-plan` | AC（受け入れ基準）を定義する | 実装前（必須） |
| `/start-feature` | 実装開始前の準備チェック | create-exec-plan 後 |
| `/run-tests` | テスト実行と仕様照合 | 実装中・完了時 |
| `/check-doc-freshness` | コードとドキュメントの乖離確認 | コード変更後 |
| `/check-invariants` | コードの INV 違反確認 | コード変更後 |
| `/check-doc-invariants` | ドキュメントの構造ルール違反確認 | ドキュメント変更後 |
| `/promote-spec` | 次バージョン仕様（`spec/<label>`）を現ターゲットへ昇格 | スプリント境界・次バージョン取り込み時 |
| `/docode-review` | 独立エージェントによるコードレビュー | 実装完了後・`/pre-pr` の前 |
| `/pre-pr` | PR 前の総合チェック | PR 作成前（必須） |
| `/complete-exec-plan` | exec-plan を完了状態に移動 | PR マージ後 |
| `/gc` | 全体的なドリフト解消 | 週次 |
