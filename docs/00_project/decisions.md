# decisions.md — アーキテクチャ決定記録（ADR）

## ADR-001: 管理スコープを「開いたフォルダ直下のみ」に限定する

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

ファイル管理の対象は、ユーザーが開いたフォルダの直下ファイルのみとする。
サブフォルダの再帰的な走査は行わない。

### 理由

- 再帰走査はファイル数が膨大になりやすく、初期ロードのパフォーマンスに悪影響が出る
- サブフォルダを含めた管理は `.filetags` の保存場所設計が複雑化する
- ユーザーは「このフォルダ内を整理したい」という局所的な意図で使用するケースが多い

### 代替案

- 再帰的走査（却下：複雑性増大）
- 深さ上限付き走査（却下：UX が直感的でなくなる）

## ADR-002: UI フレームワークに WPF + CommunityToolkit.Mvvm を採用する

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

UI フレームワークとして WPF（Windows Presentation Foundation）を使用し、
MVVM の実装には CommunityToolkit.Mvvm（旧 Microsoft.Toolkit.Mvvm）を採用する。

### 理由

- .NET 9 で WPF は公式サポート対象であり、長期的なメンテナンスが期待できる
- WinUI 3（Windows App SDK）は API の成熟度がまだ発展途上
- CommunityToolkit.Mvvm はソースジェネレーター対応でボイラープレートを削減できる
- チームの WPF 経験が豊富

### 代替案

- WinUI 3（却下：学習コスト・API 安定性の懸念）
- Avalonia（却下：クロスプラットフォームは不要）
- 素の INotifyPropertyChanged 実装（却下：記述量が多い）

## ADR-003: タグデータはフォルダ直下の `.filetags` JSON ファイルに保存する

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

タグデータ（タグ定義・ファイルとタグの紐付け）は、開いたフォルダの直下に
`.filetags` という名前の JSON ファイルとして保存する。

```json
{
  "version": 1,
  "tags": [
    { "id": "t1", "name": "重要", "color": "#FF5722" }
  ],
  "files": {
    "report.pdf": ["t1"],
    "draft.docx": ["t1"]
  }
}
```

### 理由

- フォルダを別の場所に移動・コピーしても、タグ情報が追随する（ポータビリティ）
- SQLite や外部 DB が不要でインストール・配布が簡単
- JSON は人間が直接編集・確認できる

### 代替案

- SQLite（却下：フォルダ移動でタグが失われる）
- Windows Registry（却下：ポータビリティなし）
- ファイル拡張属性（NTFS ADS）（却下：非 NTFS ドライブで機能しない）

### 注意事項

- `.filetags` ファイルは通常のファイル一覧に表示しない（隠しファイル扱い）
- ファイル名が変更・削除された場合、`files` セクションの対応エントリをクリーンアップする

## ADR-004: ファイル移動はフォルダ内のサブフォルダへの移動のみサポートする

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

一括移動操作の移動先は、現在開いているフォルダ内に存在するサブフォルダのみとする。
フォルダ外への移動はサポートしない。

### 理由

- フォルダ外への移動は `.filetags` の管理外となりタグ情報が維持できない
- 誤操作によるファイルの散逸リスクを低減する
- 移動先 UI の複雑性を抑える
