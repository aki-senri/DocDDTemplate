# decisions.md — アーキテクチャ決定記録（ADR）

## ADR-001: 管理スコープを「開いたフォルダの下位階層全体」とする

**日付**: 2026-04-16（2026-04-16 改訂）  
**状態**: 採用

### 決定内容

ファイル管理の対象は、ユーザーが開いたフォルダとその下位階層（サブフォルダを再帰的に走査）
に存在する全ファイルとする。UI はフォルダツリーとフラットなファイル一覧の両方を提供する。

### 理由

- 関連ファイルが複数のサブフォルダにまたがって存在するケースが多く、
  フォルダ直下のみでは整理のニーズを満たせない
- タグを使ってフォルダ横断的に関連ファイルを識別・移動する操作が中心ユースケース
- `.filetags` を開いたフォルダのルートに1つ置き、相対パスをキーにすることで
  サブフォルダを含めた管理を一元化できる

### 代替案

- 直下ファイルのみ（却下：ユーザー要件に合わない）
- 深さ上限付き走査（却下：UX が直感的でなくなる）

### 注意事項

- ファイル数が多い場合のパフォーマンスに注意。バックグラウンドスレッドで非同期ロードを行う
- `.filetags` のキーは相対パス（例: `"subdir/file.txt"`）を使用する（ADR-003 参照）

## ADR-002: UI フレームワークに WPF + CommunityToolkit.Mvvm を採用する

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

UI フレームワークとして WPF（Windows Presentation Foundation）を使用し、
MVVM の実装には CommunityToolkit.Mvvm（旧 Microsoft.Toolkit.Mvvm）を採用する。

### 理由

- .NET 9 で WPF は公式サポート対象であり、長期的なメンテナンスが期待できる
- WinUI 3（Windows App SDK）は API の成熟度がまだ発展途上
- CommunityToolkit.Mvvm はソースジェネレーター対応でボイラープレートを削減できる
- チームの WPF 経験が豊富

### 代替案

- WinUI 3（却下：学習コスト・API 安定性の懸念）
- Avalonia（却下：クロスプラットフォームは不要）
- 素の INotifyPropertyChanged 実装（却下：記述量が多い）

## ADR-003: タグデータはフォルダ直下の `.filetags` JSON ファイルに保存する

**日付**: 2026-04-16  
**状態**: 採用

### 決定内容

タグデータ（タグ定義・ファイルとタグの紐付け）は、開いたフォルダの直下に
`.filetags` という名前の JSON ファイルとして保存する。

```json
{
  "version": 1,
  "tags": [
    { "id": "t1", "name": "重要", "color": "#FF5722" }
  ],
  "files": {
    "report.pdf": ["t1"],
    "draft.docx": ["t1"]
  }
}
```

### 理由

- フォルダを別の場所に移動・コピーしても、タグ情報が追随する（ポータビリティ）
- SQLite や外部 DB が不要でインストール・配布が簡単
- JSON は人間が直接編集・確認できる

### 代替案

- SQLite（却下：フォルダ移動でタグが失われる）
- Windows Registry（却下：ポータビリティなし）
- ファイル拡張属性（NTFS ADS）（却下：非 NTFS ドライブで機能しない）

### 注意事項

- `.filetags` ファイルは通常のファイル一覧に表示しない（隠しファイル扱い）
- ファイル名が変更・削除された場合、`files` セクションの対応エントリをクリーンアップする

## ADR-004: ファイル移動は開いたフォルダ配下の任意フォルダへの移動をサポートする

**日付**: 2026-04-16（2026-04-16 改訂）  
**状態**: 採用

### 決定内容

一括移動操作の移動先は、現在開いているフォルダ配下に存在する任意のフォルダとする。
移動先フォルダの選択は、フォルダツリーから選択するか、パスを直接入力する方式とする。
開いたフォルダの外への移動はサポートしない。

### 理由

- 関連ファイルを整理するには、深いネスト構造のフォルダへの移動が必要になる
- 開いたフォルダ配下に限定することで `.filetags` の管理範囲が維持される
- フォルダ外への移動は `.filetags` の管理外となりタグ情報が失われる

### 代替案

- 直下サブフォルダのみ（却下：ユーザー要件に合わない）
- フォルダ外への移動を許可（却下：タグデータが失われる）

### 注意事項

- 移動後のファイルは `.filetags` 内のキー（相対パス）を新しいパスに更新する
- 移動元と移動先が同じフォルダの場合は操作をスキップする
