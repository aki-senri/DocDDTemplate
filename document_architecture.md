# AIベース開発プロジェクト ドキュメント体系

> このドキュメントはドキュメント構成の設計方針と構造を整理したものです。
> 各ドキュメントの**内容はテンプレートの型**として定義し、開発対象ごとの具体的な記述はセットアップスキルで補完します。

---

## 1. 基本方針

### 1.1 分離の原則

| 種別 | 役割 | 管理単位 |
|------|------|----------|
| **共通ドキュメント** | プロジェクト全体に適用される型・構造・ルール | このドキュメント体系 |
| **個別ドキュメント** | 開発対象（Web / Windows / API 等）に固有の内容 | セットアップスキルで生成・管理 |

### 1.2 AIとの協働を前提とした設計

- ドキュメントはAIが参照しやすい**Markdown形式**で統一する
- AIに渡す情報量を制御するため、**要約コンテキストファイル（CONTEXT.md）** を常に最新に保つ
- プロンプトテンプレートを再利用可能な形で管理し、作業品質を安定させる

### 1.3 要件の階層構造

```
ビジネスゴール
    └── 機能要件 ・ 非機能要件
            └── ユーザーストーリー（要件の具体化）
                    └── 受け入れ条件
                            └── タスク / 実装仕様
```

> **方針：トップダウンアプローチを採用**
> AIへの実装指示の精度を高めるため、機能要件・非機能要件を先に定義し、
> ユーザーストーリーはその下位レイヤーとして具体化する。

## 3. 各ディレクトリの定義

### 00_project — プロジェクト全体の文脈

| ファイル | 内容 | 共通/個別 |
|----------|------|-----------|
| `overview.md` | プロジェクトの目的・背景・スコープ | 個別 |
| `glossary.md` | 用語定義（AIの誤解防止） | 個別 |
| `decisions.md` | 重要な意思決定ログ（ADR形式） | 共通テンプレート |

---

### 01_requirements — 要件定義

#### 階層構造

```
機能要件 / 非機能要件（上位）
    └── ユーザーストーリー（下位・具体化）
            └── 受け入れ条件
```

#### ファイル構成

| ファイル | 内容 | 共通/個別 |
|----------|------|-----------|
| `functional/common.md` | 共通機能要件 | 個別 |
| `functional/{platform}.md` | プラットフォーム固有の機能要件 | 個別 |
| `non_functional/common.md` | 共通非機能要件 | 個別 |
| `non_functional/{platform}.md` | プラットフォーム固有の非機能要件 | 個別 |
| `user_stories/common.md` | 共通ユーザーストーリー | 個別 |
| `user_stories/{platform}.md` | プラットフォーム固有のストーリー | 個別 |
| `constraints.md` | 制約条件・前提条件 | 個別 |

#### プラットフォーム別の非機能要件 補足

**Windows アプリ固有で定義すべき項目**
- 対応 OS バージョン（Windows 10 / 11 等）
- インストーラ形式（MSI / MSIX / ClickOnce 等）
- オフライン動作の可否
- 自動アップデート方式
- セキュリティ要件（コード署名、UAC 対応等）
- .NET / ランタイムバージョン依存

### 06_ai_context — AI開発支援

> このディレクトリはAIとの協働品質を安定させるための中核です。

#### ファイル構成

```
06_ai_context/
├── CONTEXT.md              # AIに渡す最小コンテキスト（常に最新を維持）
├── prompts/                # 再利用可能なプロンプトテンプレート
│   ├── feature_impl.md     # 機能実装依頼
│   ├── code_review.md      # コードレビュー依頼
│   ├── test_gen.md         # テスト生成依頼
│   └── refactor.md         # リファクタリング依頼
└── skills/                 # プロジェクト固有のAIスキル定義
    ├── README.md            # スキル一覧・使い方
    └── setup/               # セットアップスキル（開発対象ごと）
        ├── web.md
        ├── windows.md
        └── api.md
```

#### CONTEXT.md に含める情報

```markdown
## CONTEXT.md の構成（テンプレート）

- プロジェクト概要（3行以内）
- 技術スタック（プラットフォーム別）
- ディレクトリ構成の要約
- 命名規則・コーディングの大原則
- 現在フェーズ・優先タスク
- 参照すべきドキュメントへのパス
```

## 4. セットアップスキルの役割と構成

### 4.1 役割の整理

```
共通ドキュメント（型）
    ＋
セットアップスキル（開発対象ごとの中身）
    ↓
実際に機能するドキュメント
```

### 4.2 セットアップスキルが担う内容

| 項目 | 例（Web） | 例（Windows） |
|------|-----------|---------------|
| 技術スタック定義 | React / Next.js / TypeScript | WPF / .NET 8 / C# |
| コーディング規約補完 | ESLint / Prettier 設定 | StyleCop / EditorConfig |
| ディレクトリ構成補完 | pages / components / hooks | Views / ViewModels / Services |
| テスト環境 | Jest / Playwright | xUnit / WinAppDriver |
| デプロイ・配布手順 | Vercel / GitHub Actions | MSIX パッケージ / WinGet |
| CI/CD 設定 | GitHub Actions (Node) | GitHub Actions (MSBuild) |

## 5. ドキュメントのライフサイクル

```
フェーズ1：プロジェクト開始
    → 00_project, 01_requirements（共通部分）を作成
    → セットアップスキルを選択・実行して個別部分を補完

フェーズ2：設計
    → 02_design を作成
    → CONTEXT.md を初版作成

フェーズ3：実装開始
    → 03_implementation をスキルで補完
    → prompts/ テンプレートを使ってAIに実装を依頼

フェーズ4：品質管理
    → 04_quality のチェックリストを使ってレビュー・テスト

フェーズ5：リリース・運用
    → 05_operations を参照してデプロイ
    → decisions.md に意思決定を記録

継続的：
    → CONTEXT.md を常に最新に保つ
    → decisions.md に変更の経緯を蓄積
```

## 6. 未決事項・今後の検討ポイント

| 項目 | 内容 | 優先度 |
|------|------|--------|
| セットアップスキルの粒度 | 言語・フレームワーク単位か、プロジェクト種別単位か | 高 |
| ドキュメントのバージョン管理 | Git管理前提か、別途バージョニングが必要か | 中 |
| CONTEXT.md の更新タイミング | 手動か、ドキュメント変更をトリガーにするか | 中 |
| 複数プラットフォーム共存時の設計方針 | モノレポ構成かリポジトリ分割か | 高 |
| テンプレートの管理場所 | このリポジトリ内か、別リポジトリか | 低 |
