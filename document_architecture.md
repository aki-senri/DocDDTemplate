# AIベース開発プロジェクト ドキュメント体系

> このドキュメントはドキュメント構成の設計方針と構造を整理したものです。
> 開発のドキュメントの指針を示すファイルを作成するのが目的です。
> 作成が完了した時点でこのファイルの役割は終わります。
> 終了条件は、
> 1. ドキュメントの参照ルールを示すファイルを用意すること。
> 2. ドキュメントの構成と内容の指針を示すファイルを用意すること。
> 3. ドキュメントの構成と内容の指針をセットアップスキルで補完すること。
> 4. 各ドキュメントのテンプレートを各工程用のスキルに用意すること。
> 最終的にこのプロジェクトは、そのまま他のプロジェクトに展開できるドキュメント運用関連のテンプレート集になることを目指します。

---

## 1. 基本方針

### 1.1 分離の原則

| 種別 | 役割 | 管理単位 |
|------|------|----------|
| **共通ドキュメント** | プロジェクト全体に適用される型・構造・ルール | このドキュメント体系 |
| **個別ドキュメント** | 開発対象（Web / Windows / API 等）に固有の内容 | セットアップスキルで生成・管理 |

### 1.2 AIとの協働を前提とした設計

- ドキュメントはAIが参照しやすい**Markdown形式**で統一する
- AIに渡す情報量を制御するため、**CONTEXT.md はナビゲーションマップ**として設計する（百科事典にしない）
- エージェントは**リポジトリ内にあるものしか認識できない**。Slack・口頭合意・外部ドキュメントの内容はリポジトリに書き起こす
- プロンプトテンプレートを再利用可能な形で管理し、作業品質を安定させる

> **CONTEXT.md 設計原則**
> 「1000ページの取扱説明書」ではなく「地図」として機能させる。
> 何が重要かではなく「どこを見ればわかるか」を記述し、最大1画面に収める。
> コンテキスト枯渇を防ぐため、詳細は各ドキュメントへのポインタで示す。

### 1.3 要件の階層構造

```
ビジネスゴール
    └── 機能要件 ・ 非機能要件
            └── ユーザーストーリー（要件の具体化）
                    └── 受け入れ条件
                            └── タスク / 実装仕様
```

> **方針：トップダウンアプローチを採用**
> AIへの実装指示の精度を高めるため、機能要件・非機能要件を先に定義し、
> ユーザーストーリーはその下位レイヤーとして具体化する。

### 1.4 アーキテクチャの不変条件を機械的に強制する

- ドキュメントだけでは一貫性を保てない。リンターとCI により不変条件を強制する
- 依存方向・ファイルサイズ・命名規則などを `03_implementation/invariants.md` で定義し、カスタムリンターに落とし込む
- リンターのエラーメッセージに修復手順を含めることで、エージェントが自己修正できるようにする

### 1.5 ガベージコレクションで技術的負債を継続的に解消する

- エージェントはリポジトリ内の既存パターンを複製するため、時間とともにドリフトが生じる
- 定期的なガベージコレクションエージェントを運用し、ドキュメント陳腐化・アーキテクチャドリフトを自動検出・修正する
- 技術的負債は高利のローン。複利で膨らむ前に常に少しずつ返済し続ける

### 03_implementation — 実装ガイド

> **共通の型として定義し、中身はセットアップスキルで補完する**

| ファイル | 内容 | 共通/個別 |
|----------|------|-----------|
| `coding_standards.md` | コーディング規約 | 共通テンプレート → スキルで補完 |
| `directory_structure.md` | コードの配置ルール | 共通テンプレート → スキルで補完 |
| `patterns.md` | デザインパターン・慣習 | 共通テンプレート → スキルで補完 |
| `dependencies.md` | 外部ライブラリ・バージョン管理方針 | 個別 |
| `invariants.md` | 機械的に強制する不変条件の定義 | 共通テンプレート → スキルで補完 |

#### invariants.md に定義する不変条件の例

- レイヤー依存方向（例：型 → 設定 → リポジトリ → サービス → UI への一方向のみ許可）
- ファイルサイズ上限
- 命名規則（スキーマ・型・ファイル名）
- 境界でのデータ検証必須（未検証データをそのまま内部に流さない）
- 構造化ロギング必須
- プラットフォーム固有の信頼性要件

> **運用方針**：不変条件はカスタムリンターと構造テストで CI に組み込む。
> リンターのエラーメッセージには修復手順を記載し、エージェントが自己修正できるようにする。

### 05_operations — 運用・デプロイ

| ファイル | 内容 | 共通/個別 |
|----------|------|-----------|
| `environments.md` | 環境構成（dev / staging / prod） | 共通テンプレート → スキルで補完 |
| `deployment.md` | デプロイ・配布手順 | 個別 |
| `monitoring.md` | 監視・ログ方針・ガベージコレクション運用 | 共通テンプレート |

#### monitoring.md に含めるガベージコレクション運用手順

- **ドキュメント整備エージェント**：実コードと乖離したドキュメントをスキャンし、修正PRを作成
- **アーキテクチャドリフト検出エージェント**：`invariants.md` 定義の違反を検出し、リファクタリングPRを作成
- **CONTEXT.md 更新エージェント**：フェーズ・優先タスクを常に最新に保つ
- 実行頻度：週次（または大きな機能マージ後）
- 生成されたPRのレビュー所要時間を1分以内に抑えるよう設計する

### 06_ai_context — AI開発支援

> このディレクトリはAIとの協働品質を安定させるための中核です。

#### ファイル構成

```
06_ai_context/
├── CONTEXT.md              # AIに渡すナビゲーションマップ（常に最新・最大1画面）
├── prompts/                # 再利用可能なプロンプトテンプレート
│   ├── feature_impl.md     # 機能実装依頼
│   ├── code_review.md      # コードレビュー依頼
│   ├── test_gen.md         # テスト生成依頼
│   └── refactor.md         # リファクタリング依頼
└── skills/                 # プロジェクト固有のAIスキル定義
    ├── README.md            # スキル一覧・使い方
    └── setup/               # セットアップスキル（開発対象ごと）
        ├── web.md
        ├── windows.md
        └── api.md
```

#### CONTEXT.md に含める情報

```markdown
## CONTEXT.md の構成（ナビゲーションマップ）

- プロジェクト概要（3行以内）
- 技術スタック（プラットフォーム別）
- 開発ルール（プロジェクト固有の合意事項）
  - ブランチ運用
  - PR・レビュー方針
  - テスト方針
  - AIエージェント活用範囲
- ドキュメント構成の要約（各ドキュメントへのパス付き）
- 命名規則・コーディングの大原則（詳細は invariants.md へ）
- 現在フェーズ・優先タスク（exec-plans/active/ へのリンク）
- 参照すべきドキュメントへのパス
```

> **「開発ルール」の記載方針**：ブランチ戦略・レビュー方針・テスト方針・AI活用範囲は
> プロジェクトごとに異なるため、CONTEXT.md に直接記載する。
> 詳細な規約（コーディング規約等）は別ドキュメントに書いてポインタで参照する。

> **更新ルール**：フェーズ移行・優先タスク変更のたびに更新する。
> 内容が膨らんだら詳細を別ドキュメントに切り出してポインタに置き換える。

> 進行中・完了済みの計画を第一級成果物としてバージョン管理する。
> `decisions.md`（過去の意思決定ログ）とは異なり、**現在進行形の計画・進捗・判断ログ**を管理する。

#### ファイル構成

```
exec-plans/
├── active/
│   └── YYYY-MM-{feature-name}.md   # 進行中の計画
└── completed/
    └── YYYY-MM-{feature-name}.md   # 完了済みの計画（履歴として保持）
```

#### 各計画ファイルに含める情報

```markdown
## exec-plans/{name}.md の構成

- 目標・スコープ（3行以内）

- タスクの分解（チェックリスト形式）
- 進捗ログ（日付付き）
- 判断ログ（なぜそうしたかの記録）
- ステータス：[active|completed|abandoned]
```

### 4.1 役割の整理

## 5. 開発ワークフロー

### フェーズ0：リポジトリ初期化（エージェントファースト設計）

```
目的：エージェントが最初の一手を打てる最小環境を作る

成果物：
├── AGENTS.md（~100行のナビゲーションマップ）
├── 06_ai_context/CONTEXT.md（初版）
└── CI骨格（lint / test）
```

- セットアップスキルを選択・実行して技術スタックを確定する
- 00_project/overview.md と constraints.md を先に作成する（制約を最初に固定）

---

### フェーズ1：ナレッジベース構築

```
目的：エージェントがリポジトリ自体から推論できる状態を作る

成果物：
├── 00_project/（overview, glossary）
├── 01_requirements/constraints.md
└── 03_implementation/invariants.md（初版）
```

- 外部で合意した内容（アーキテクチャ方針・チーム文化・設計原則）をリポジトリに書き起こす
- 各ドキュメントに `status:` フィールドを付与する
- CI にドキュメント鮮度チェックを追加する

---

### フェーズ2：要件定義・設計

```
目的：トップダウンで意図を明確化し、実行計画をバージョン管理する

成果物：
├── 01_requirements/（functional, non_functional, user_stories）
├── 02_design/architecture.md
└── exec-plans/active/{feature}.md（計画を先にリポジトリへ）
```

- CONTEXT.md を更新してフェーズ移行を反映する

---

### フェーズ3：エージェント駆動の実装ループ

```
目的：エージェントが自律的に実装→検証→PRを回せる状態を維持する

ループ：
1. プロンプト作成（prompts/ テンプレートを使用）
2. エージェントが実装 → テスト → セルフレビュー → PR作成
3. アーキテクチャリンター（invariants）通過を必須条件とする
4. エージェント or 人間がレビュー → マージ
   └── ブロックを最小化（フレーキーなテストはフォローアップで対処）
```

- 03_implementation/ をスキルで補完する
- exec-plans/active/ の進捗ログを更新し続ける
- 完了した計画は completed/ に移動する

---

### フェーズ4：継続的品質維持（ガベージコレクション）

```
目的：エージェントが生成したドリフトを自動的に解消し続ける

定期エージェントタスク：
├── ドキュメント整備（実コードと乖離を検出 → PR作成）
├── アーキテクチャドリフト修正（リンター違反を検出 → リファクタリングPR）
└── CONTEXT.md 更新（フェーズ・優先タスクを最新に保つ）
```

- 04_quality/ のチェックリストを使ってレビュー・テスト
- decisions.md に変更の経緯を蓄積する

---

### 継続的

```
→ CONTEXT.md を常に最新に保つ（ナビゲーションマップとして機能し続けること）
→ decisions.md に意思決定を記録
→ exec-plans/ を計画の唯一の真実（Single Source of Truth）として維持
→ ガベージコレクションエージェントを定期実行
```

## 6. スキルの標準フォーマット定義

> すべてのスキルファイルはこのフォーマットに従う。
> フォーマットに従うことで、ガベージコレクションエージェントが「スキルに定義されたファイルが実際に存在するか」を自動検証できる。

### 6.1 フロントマター（必須）

```yaml
---
name: skill-name          # ケバブケース。スキル呼び出し時の識別子
description: |            # いつ使うかを1〜2行で説明
  ...
disable-model-invocation: true   # Claude が勝手に呼び出さないようにする場合は true
---
```

### 6.2 必須セクション

| セクション | 内容 |
|-----------|------|
| `## 実行タイミング` | いつ・何をきっかけに使うスキルかを明示 |
| `## このスキルが作るもの / このスキルがすること` | 成果物一覧（ファイルパス・役割・テンプレート参照） |
| `## 手順` | コマンドや確認ステップをリスト形式で記述 |
| `## 完了条件` | チェックリスト形式。エージェントが完了判定に使う |

### 6.3 成果物一覧の記述形式

生成ファイルがある場合は以下の表形式で記述する。

```markdown
| ファイルパス | 役割 | テンプレート参照 |
|-------------|------|----------------|
| `docs/xxx/yyy.md` | ～の定義 | §テンプレート名 または インライン |
```

### 6.4 コードテンプレートの埋め込み

初期テンプレートが必要なファイルは、スキル内にインラインで含める。

````markdown
#### `{ファイルパス}` のテンプレート

```{言語}
// テンプレートの内容
```
````

## 7. ドキュメントの必須/任意区別と分割基準

### 7.1 必須/任意区別

各ドキュメントに `必須/任意` を定義する。任意ファイルはそのファイルが必要になる条件を明記する。

| ファイル | 必須/任意 | 必要な条件 |
|---------|---------|----------|
| `00_project/overview.md` | 必須 | 常に |
| `00_project/glossary.md` | 任意 | 用語の誤解リスクがある場合 |
| `00_project/decisions.md` | 必須 | 常に（技術選定ADRを記録） |
| `01_requirements/functional/common.md` | 任意 | 複数プラットフォームが存在する場合 |
| `01_requirements/functional/{platform}.md` | 必須 | 常に（プラットフォームごとに1ファイル） |
| `01_requirements/non_functional/common.md` | 任意 | 複数プラットフォームが存在する場合 |
| `01_requirements/non_functional/{platform}.md` | 必須 | 常に |
| `01_requirements/user_stories/common.md` | 任意 | 複数プラットフォームが存在する場合 |
| `01_requirements/user_stories/{platform}.md` | 必須 | 常に |
| `01_requirements/constraints.md` | 必須 | 常に |
| `02_design/architecture.md` | 必須 | 常に |
| `02_design/data_model.md` | 任意 | DBや永続化ストレージを持つ場合 |
| `02_design/api_spec.md` | 任意 | 外部公開APIを持つ場合 |
| `02_design/ui_flows.md` | 任意 | UIを持つ場合 |
| `03_implementation/coding_standards.md` | 必須 | 常に |
| `03_implementation/directory_structure.md` | 必須 | 常に |
| `03_implementation/patterns.md` | 必須 | 常に |
| `03_implementation/dependencies.md` | 必須 | 常に |
| `03_implementation/invariants.md` | 必須 | 常に |
| `04_quality/test_strategy.md` | 必須 | 常に |
| `04_quality/review_checklist.md` | 必須 | 常に |
| `04_quality/security.md` | 任意 | 外部通信・認証・機密データを扱う場合 |
| `04_quality/performance.md` | 任意 | パフォーマンス要件が明示されている場合 |
| `05_operations/environments.md` | 任意 | dev/staging/prod の環境分離が必要な場合 |
| `05_operations/deployment.md` | 必須 | 常に |
| `05_operations/monitoring.md` | 必須 | 常に |
| `06_ai_context/CONTEXT.md` | 必須 | 常に |

### 7.2 common/platform 分割基準

| 状況 | ルール |
|------|--------|
| プラットフォームが1つだけ | `common.md` は作成しない。すべて `{platform}.md` に記述する |

## 8. ドキュメントのライフサイクル管理

### 8.1 `status:` フィールド

すべてのドキュメントのフロントマターに `status:` を記述する。

```yaml
---
status: draft    # draft | active | deprecated
tracks:          # （任意）このドキュメントが対応するコード資産（glob パターン）
  - src/**/models/**
---
```

### 8.2 `status:` 遷移ルール

| 遷移 | トリガー |
|------|---------|
| （新規）→ `draft` | ファイル作成時。内容が未確定の場合 |
| `draft` → `active` | PRマージ時。または作成から7日以内にレビューを受けた場合 |
| `active` → `deprecated` | 対応するコード/機能が削除された場合。または後継ドキュメントが `active` になった場合 |
| `deprecated` → （削除） | `deprecated` から30日経過後に削除可 |

### 8.3 `tracks:` フィールド（doc-to-codeマッピング）

`tracks:` フィールドに glob パターンを記述することで、ドキュメントとコード資産の対応関係を定義する。
パスパターンはプロジェクトの言語・ディレクトリ構成に応じて記述する。

```yaml
---
status: active
tracks:
  - src/**/models/**          # プロジェクトの言語・構成に応じたパスを記述する
  - src/**/repositories/**
---
```

**運用ルール：**
- `check-doc-freshness` スキルは `tracks:` フィールドを読み取り、変更されたコードファイルに対応するドキュメントを特定する
- `gc` スキルは `tracks:` フィールドを使ってドキュメント鮮度の全件スキャンを行う
- `tracks:` がないドキュメントはGCの対象外となる

**`tracks:` を設定すべきドキュメントと、パスパターンの例：**

| ドキュメント | tracks パターンの例（言語・構成に応じて変更する） |
|------------|----------------------------------------------|
| `data_model.md` | `src/**/models/**`, `src/**/entities/**` |
| `api_spec.md` | `src/**/controllers/**`, `src/**/routes/**`, `src/**/api/**` |
| `invariants.md` | `src/**` |
| `coding_standards.md` | `src/**` |
| `directory_structure.md` | `src/**`, `frontend/**`, `backend/**` |

## 9. exec-plans のライフサイクル管理

### 9.1 ステータス遷移

| 状態 | 説明 | 配置場所 |
|------|------|---------|
| `active` | 進行中 | `exec-plans/active/` |
| `completed` | 完了済み | `exec-plans/completed/` |
| `abandoned` | 中断・破棄 | `exec-plans/completed/`（ファイル内に理由を記録） |

### 9.2 各計画ファイルの構成（テンプレート）

```markdown
---
status: active
created: YYYY-MM-DD
completed:           # 完了日（完了時に記入）
---

# {機能名・タスク名}

## 目標・スコープ
（3行以内）

## 受け入れ条件
- [ ] 条件1
- [ ] 条件2

## タスク分解
- [ ] タスク1
- [ ] タスク2

## 進捗ログ
### YYYY-MM-DD
- 実施内容

## 判断ログ
### YYYY-MM-DD
- 判断内容と理由
```

### 9.3 CONTEXT.md との連動ルール

| exec-plans の変化 | CONTEXT.md の更新内容 |
|-----------------|-------------------|
| 新規 `active/` ファイル作成 | 優先タスクを更新 |
| `active/` → `completed/` 移動 | 現在フェーズ・優先タスクを更新 |
| フェーズ移行（Phase 1→2 等） | 現在フェーズを更新 |
