# テンプレート共通ギャップ分析

> このドキュメントは `document_architecture.md` の定義と、実際のサンプルドキュメント作成（FocusKeeper: Windowsデスクトップアプリ）を通じて発見した**共通の不足点**をまとめたものです。
> プラットフォーム固有スキル（`06_ai_context/skills/setup/`）の整備に先立ち、共通設計として解決すべき課題を記述します。

---

## 発見の契機

Windowsアプリのサンプルドキュメントを作成する中で、**ドキュメントの内容自体は定義に沿って書けるが、エージェントが実際に開発を開始できる状態にはならない**という問題が明確になりました。

根本原因：**セットアップスキルが「手順の説明」にとどまっており、エージェントが直接利用できるコード構造テンプレートを含んでいない。**

以下に、特定プラットフォームに依存しない共通の不足点を列挙します。

---

## GAP-01: セットアップスキルにコード構造テンプレートが未統合

### 現状

`document_architecture.md` のセットアップスキル定義（§4）は「何を担うか」の表形式説明のみ。
`06_ai_context/skills/setup/{platform}.md` に書くべき内容の**フォーマットが未規定**であるため、
サンプルで作成した `skills/setup/windows.md` は手順の散文になった。

```
現状のスキルの構造（実態）:
  前提条件
  → Step 1: ツールインストール確認
  → Step 2: クローン
  → Step 3: ビルド確認
  → ...

エージェントが必要とする構造:
  前提条件
  → 生成すべきファイル一覧（パス付き）
  → 各ファイルの初期テンプレート（コピー可能なコード）
  → セットアップ完了の検証手順
```

### 影響

- エージェントはスキルを読んでも「どのファイルを何という内容で作ればよいか」がわからない
- 開発者がスキルを参照してゼロから書き起こす必要があり、AIによる自律セットアップが機能しない
- プラットフォームごとにスキルの書き方が属人化する

### 対応方針

**セットアップスキルの標準フォーマットを定義し、そのフォーマット内にコード構造テンプレートを埋め込む。**
（→ 後述の GAP-02 で標準フォーマット自体を定義する）

---

## GAP-02: スキルが生成すべき成果物の定義フォーマットが未規定

### 現状

スキルが「何を生成するか」の仕様（成果物リスト・テンプレート・検証方法）を記述する
標準的なフォーマットが `document_architecture.md` に存在しない。

各スキルを書く人・エージェントが構造を自由に決めることになるため、一貫性が失われる。

### 影響

- スキルAとスキルBで記述スタイルが異なると、エージェントがスキルを解釈する際にパースコストが上がる
- ガベージコレクションエージェントが「スキルに定義されたファイルが実際に存在するか」を検証できない
- スキルの充実度（テンプレートが含まれているか否か）を外から判断できない

### 対応方針

セットアップスキルの標準フォーマット（以下を必須セクションとして定義する）を `document_architecture.md` に追記する。

```markdown
## スキルの標準フォーマット（案）

### メタ情報
- 対象プラットフォーム
- 前提条件（ツール・権限）

### 生成ファイル一覧
| パス | 役割 | テンプレート参照 |
|------|------|----------------|
| ...  | ...  | §テンプレート名  |

### コードテンプレート
#### {ファイル名}
```{言語}
// 初期テンプレートの内容
```

### セットアップ手順
（コマンドによる実行手順）

### 完了検証チェックリスト
- [ ] 検証項目1
- [ ] 検証項目2
```

## GAP-03: 必須ドキュメントと任意ドキュメントの区別が未定義

### 現状

`document_architecture.md` §3 で定義されているすべてのファイルが、同じ記述レベルで並んでいる。
プロジェクトの性質によっては不要なファイルも存在する。

**不要になりうるファイルの例：**

| ファイル | 不要になるケース |
|---------|----------------|
| `02_design/api_spec.md` | デスクトップアプリ・CLIツールなどAPIを持たない場合 |
| `01_requirements/functional/common.md` | 単一プラットフォームのみのプロジェクト |
| `05_operations/environments.md` | ローカル完結型でdev/staging/prod分離がない場合 |

### 影響

- エージェントが「全ファイルを作成すべき」と判断し、`api_spec.md` に「該当なし」と書くだけのファイルを作成する
- または不要なファイルを省略するかどうかで迷い、判断が分かれる
- ドキュメント鮮度チェックが「空のファイル」を対象にして誤検知を起こす

### 対応方針

各ドキュメント定義に `必須/任意` の属性を追加する。
任意ファイルは「そのファイルが必要な条件」を明記する。

```markdown
## 定義フォーマット（改定案）

| ファイル | 内容 | 必須/任意 | 必要な条件 |
|---------|------|---------|----------|
| `api_spec.md` | API仕様 | 任意 | 外部公開APIを持つ場合 |
| `functional/common.md` | 共通機能要件 | 任意 | 複数プラットフォームが存在する場合 |
```

## GAP-04: functional/common.md と platform固有ファイルの分割基準が未定義

### 現状

`document_architecture.md` §3 は `functional/common.md` と `functional/{platform}.md` の
**両方が存在することを前提**としているが、どちらに何を書くかの判断基準がない。

### 影響

- 単一プラットフォームのプロジェクトで `common.md` を作るべきか迷う
- 複数プラットフォームがある場合、「プラットフォーム共通の要件」の定義が人によって異なる
- エージェントが要件を参照する際、同じ要件が `common.md` と `windows.md` に重複して書かれるリスク

### 対応方針

分割基準を明文化する。

```markdown
## common.md / platform固有ファイルの分割ルール（案）

- プラットフォームが1つだけ → common.md は作成しない。すべて {platform}.md に書く
- プラットフォームが2つ以上 → 全プラットフォームに共通する要件を common.md に記述
- 「共通だが実装方法が違う」要件 → common.md に要件を記述し、各 {platform}.md で実装制約を補記
```

## GAP-05: status フィールドの遷移トリガーが未定義

### 現状

`document_architecture.md` §4 の品質管理節で `status: [draft|active|deprecated]` の存在が示されているが、
**どのタイミングで状態を変えるかのルールがない**。

### 影響

- ドキュメントが `status: draft` のまま放置される
- `deprecated` への移行タイミングが属人的になり、GCエージェントが古いドキュメントを検出できない
- `test_strategy.md` に「draftが14日以上経過で警告」と記述したが（サンプル内）、この数字に根拠がない

### 対応方針

ライフサイクルのトリガーを定義する。

```markdown
## status: 遷移ルール（案）

| 遷移 | トリガー |
|------|---------|
| （新規）→ draft | ファイル作成時。内容が未確定の場合 |
| draft → active | PRマージ時。または作成から7日以内にレビューを受けた場合 |
| active → deprecated | 対応するコード/機能が削除された場合。または後継ドキュメントが active になった場合 |
| deprecated → （削除） | deprecated から30日経過後に削除可 |
```

## GAP-06: ドキュメントとコードの対応マッピングが未定義

### 現状

ガベージコレクション運用（§3 `monitoring.md`）では「実コードと乖離したドキュメントを検出する」と定義しているが、
**どのドキュメントがどのコード資産に対応するか**のマッピング規則がない。

### 影響

- GCエージェントが「全コード vs 全ドキュメント」を網羅的にスキャンするしかなく、実行コストが高い
- 「`data_model.md` のDDLとDBスキーマが乖離した場合に `data_model.md` を更新する」という判断ができない
- コードを変更したエージェントが、どのドキュメントを更新すべきかを自律的に判断できない

### 対応方針

ドキュメントごとに「乖離検出の対象コード資産」を明記する規則を設ける。

```markdown
## ドキュメント対応マッピング（doc-to-code マッピング）の記述規則（案）

各ドキュメントのヘッダに以下を追記する:

---
status: active
tracks:
  - src/**/Models/*.cs          # data_model.md の場合
  - src/**/Repositories/*.cs    # data_model.md の場合
---

この `tracks:` フィールドをGCエージェントが読み取り、対象ファイルの変更時に
ドキュメントの鮮度チェックをトリガーする。
```

## 対応優先度まとめ

| GAP | タイトル | 優先度 | 備考 |
|-----|---------|--------|------|
| GAP-01 | スキルにコード構造テンプレートが未統合 | **最高** | スキル整備の前提条件 |
| GAP-02 | スキルの標準フォーマットが未規定 | **最高** | GAP-01 の解決に必要 |
| GAP-03 | 必須/任意ドキュメントの区別が未定義 | 高 | エージェントの迷いを防ぐ |
| GAP-04 | common/platform 分割基準が未定義 | 高 | 複数プラットフォーム対応前に必要 |
| GAP-05 | status 遷移トリガーが未定義 | 中 | GCエージェント精度に影響 |
| GAP-06 | doc-to-code マッピングが未定義 | 中 | GCエージェントの実装時に必要 |

---

## 次のアクション

1. **GAP-01 / GAP-02 を解消する：セットアップスキルの標準フォーマット定義を `document_architecture.md` に追記する**  
   → これが完了すれば、各プラットフォームのスキル整備（`skills/setup/windows.md` 等）を本格的に進められる

2. **GAP-03 / GAP-04 を解消する：ドキュメント定義テーブルに `必須/任意` 列と分割基準を追記する**  
   → スキルがドキュメント生成テンプレートを内包するため、どのファイルを生成すべきかの判断基準として必要

3. GAP-05 / GAP-06 はフェーズ4（ガベージコレクション）の実装時に対応する
